## 5. 踩过的坑 / 经验总结

### 坑 1：OnStepX GPIO36 无法使用 INPUT_PULLUP
- **症状**：固件启动串口报 `gpio_pullup_en(85): GPIO number error (input-only pad has no internal PU)`，设备可能不稳定
- **原因**：ESP32 GPIO36 是 input-only 引脚，没有内部上拉电阻；DS3231 的 SQW/INT 输出是 open-drain，板子上有外部上拉，不需要内部 PULLUP
- **解决**：`Pins.MaxESP3.h` 中定义 `PPS_SENSE_INIT INPUT`，`PPS.cpp` 中改用 `PPS_SENSE_INIT` 而不是硬编码 `INPUT_PULLUP`
- **预防**：ESP32 各 GPIO 的能力差异大，修改引脚相关代码前先查 ESP32 datasheet 或原理图

### 坑 2：OnStepX GitHub Actions tag 触发规则不完整
- **症状**：push 纯数字 tag（如 `1.0.1`）不触发 release Actions，带字母后缀（如 `1.0.1a`）才触发
- **原因**：`.github/workflows/build-release.yml` 的 tag pattern 是 `[0-9]*.[0-9]*.[0-9]*[A-Za-z]*`，强制要求末尾有字母
- **解决**：将 pattern 改为 `[0-9]*.[0-9]*.[0-9]*` 或 `v[0-9]*` 即可
- **预防**：设置 Actions 触发规则后，用不同格式的 tag 各测一次

### 坑 3：INDIGO UTC Time 每秒轮询覆盖用户输入
- **症状**：在 indigo web 的 Mount LX200 → Site → UTC Time 输入时间后，过一会儿时间自动被覆盖
- **原因**：INDIGO 框架每秒从本机系统时钟读取当前时间并推送到 UTC_TIME 属性，用户手动输入的值会立即被覆盖
- **解决**：实际使用时不需要修改 UTC Time（相机系统时间是准确的），只需要修改 UTC Offset 来适配不同时区
- **预防**：理解 UTC Time 字段的实际含义是"从 host 推给 OnStep 赤道仪的时间"，不是"相机自身的系统时间"

### 坑 4：stargate 在 seeing 中作为 submodule
- **症状**：修改了 `stargate` 并 push，但 `seeing` 项目中的 stargate 还是旧版本
- **原因**：`seeing` 以 submodule 方式引用 `stargate`，stargate 更新后 seeing 的 submodule 指针要手动更新
- **解决**：在 `seeing` 目录下 `cd stargate && git pull && cd .. && git add stargate && git commit -m "update stargate submodule"`
- **预防**：每次 stargate 有重要更新后，记得同步 seeing 的 submodule 指针

### 通用经验

- 💡 stargate 交叉编译通过 `seeing` 项目完成，编译结果在 `seeing/output/`，刷机方式参考 `seeing/README_CN.md`
- 💡 indigo_server 的 log 在本机运行时输出到 stdout，用 `indigo_server -vvv 2>&1 | tee indigo.log` 保存；trace 级别日志量极大，debug 时配合 grep 筛选
