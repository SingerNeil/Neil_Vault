## 3. 已完成的工作沉淀

### 3.1 Pulsar 赤道仪 OTA 固件升级系统

这是跨越三个仓库的核心功能，允许通过 USB 串口对 Pulsar 赤道仪的两块 MCU 进行无线/串口固件升级，并在 INDIGO web 界面中提供操作入口。

**系统架构：**

```
INDIGO web 界面 (stargate)
    ↓ 触发 OTA 流程
indigo_mount_lx200_serial_ota.c
    ↓ USB-C 串口（LX200 协议）
MCU1 = OnStepX (主控板, OnStepX.ino.bin)
    ↓ CN2 串口透传
MCU2 = SmartWebServer (WiFi 模块, SmartWebServer.ino.bin)
```

**升级命令协议（命令格式定义在 `SmartWebServer/UPGRADE_OFFICIAL_NAMING_SPEC.md`）：**

| 命令 | 含义 |
|------|------|
| `:FL00#` | 请求刷写 MCU1 |
| `:FL01#` | 请求刷写 MCU2（经由 MCU1 透传） |
| `FR00#` | MCU1 就绪 |
| `FP01#` | MCU1 透传通道就绪 |
| `:FU01#` | 唤醒 MCU2 进入 OTA 接收模式 |
| `FR01#` | MCU2 就绪 |

**二进制帧格式（START/DATA/END/ACK/NACK）：**
```
[0x55][0xAA][CMD 1B][SEQ 2B][LEN 2B][PAYLOAD][CRC16 2B]
```
- MCU1 升级：握手后切换至 **460800** 波特率传输（Linux only，macOS 不支持）
- MCU2 升级：全程保持 **115200**，经 MCU1 CN2 串口透传

**各仓库中的相关文件：**

| 仓库 | 文件 | 说明 |
|------|------|------|
| stargate | `indigo_drivers/mount_lx200/indigo_mount_lx200_serial_ota.c/.h` | INDIGO 侧 OTA 主逻辑（串口协议实现） |
| stargate | `indigo_drivers/mount_lx200/indigo_mount_lx200_ota.c` | CAN 总线 OTA（CAA 旋转器升级） |
| stargate | `indigo_drivers/mount_lx200/indigo_mount_lx200.c` | OTA UI 属性注册、进度显示 |
| SmartWebServer | `src/pages/update/OtaReceiver.cpp` | MCU2 接收固件数据 |
| SmartWebServer | `src/pages/update/OtaForwarder.cpp` | MCU2 透传转发给 MCU1 |
| SmartWebServer | `src/pages/update/OtaTrap.cpp` | MCU2 进入 OTA 等待模式 |
| OnStepX | `src/` 下各 OTA 相关文件 | MCU1 接收固件并写 flash |
| OnStepX | `onstepx_upgrade.py` | 上位机 Python 升级工具（串口命令行） |

**OTA 流程文档：** `stargate/ONSTEPX_OTA_PROTOCOL_CN.md` 和 `ONSTEPX_OTA_PROTOCOL.md`

**⚠️ 注意事项：**
- OTA 过程中断电会导致 MCU 变砖，需要用 USB 下载器救砖
- 460800 波特率仅 Linux 支持，macOS 环境无法刷 MCU1
- OTA 进度 UI 只在进度百分比变化时更新（PR #49 优化），避免频繁刷新 INDIGO 总线

## 5. 踩过的坑 / 经验总结

### 坑 1：OnStepX GPIO36 无法使用 INPUT_PULLUP
- **症状**：固件启动串口报 `gpio_pullup_en(85): GPIO number error (input-only pad has no internal PU)`，设备可能不稳定
- **原因**：ESP32 GPIO36 是 input-only 引脚，没有内部上拉电阻；DS3231 的 SQW/INT 输出是 open-drain，板子上有外部上拉，不需要内部 PULLUP
- **解决**：`Pins.MaxESP3.h` 中定义 `PPS_SENSE_INIT INPUT`，`PPS.cpp` 中改用 `PPS_SENSE_INIT` 而不是硬编码 `INPUT_PULLUP`
- **预防**：ESP32 各 GPIO 的能力差异大，修改引脚相关代码前先查 ESP32 datasheet 或原理图

### 坑 2：OnStepX GitHub Actions tag 触发规则不完整
- **症状**：push 纯数字 tag（如 `1.0.1`）不触发 release Actions，带字母后缀（如 `1.0.1a`）才触发
- **原因**：`.github/workflows/build-release.yml` 的 tag pattern 是 `[0-9]*.[0-9]*.[0-9]*[A-Za-z]*`，强制要求末尾有字母
- **解决**：将 pattern 改为 `[0-9]*.[0-9]*.[0-9]*` 或 `v[0-9]*` 即可
- **预防**：设置 Actions 触发规则后，用不同格式的 tag 各测一次

### 坑 3：INDIGO UTC Time 每秒轮询覆盖用户输入
- **症状**：在 indigo web 的 Mount LX200 → Site → UTC Time 输入时间后，过一会儿时间自动被覆盖
- **原因**：INDIGO 框架每秒从本机系统时钟读取当前时间并推送到 UTC_TIME 属性，用户手动输入的值会立即被覆盖
- **解决**：实际使用时不需要修改 UTC Time（相机系统时间是准确的），只需要修改 UTC Offset 来适配不同时区
- **预防**：理解 UTC Time 字段的实际含义是"从 host 推给 OnStep 赤道仪的时间"，不是"相机自身的系统时间"

### 坑 4：stargate 在 seeing 中作为 submodule
- **症状**：修改了 `stargate` 并 push，但 `seeing` 项目中的 stargate 还是旧版本
- **原因**：`seeing` 以 submodule 方式引用 `stargate`，stargate 更新后 seeing 的 submodule 指针要手动更新
- **解决**：在 `seeing` 目录下 `cd stargate && git pull && cd .. && git add stargate && git commit -m "update stargate submodule"`
- **预防**：每次 stargate 有重要更新后，记得同步 seeing 的 submodule 指针
