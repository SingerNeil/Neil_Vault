**📝 背景与复现步骤**：
1. 相机断开外部网络（无 NTP 同步，系统时钟停在 epoch = 1970-01-01）
2. 电脑连接相机自身的 WiFi（如 myAP）
3. INDIGO → Mount LX200 → Control Panel → Site → 点击 **Sync System from Mount**
4. 操作失败，报错：`Failed to set system time (requires root)`
5. 同时 syslog 有：`chronyd[390]: Could not open /var/lib/chrony/drift.tmp: Permission denied`

**🔍 根因分析**：

两个独立问题交织：

**问题 A（主要）：indigo_server 没有 root 权限**
`Sync System from Mount` 最终调用 `clock_settime(CLOCK_REALTIME, ...)` 来把赤道仪时间写回相机系统时钟。这个系统调用需要 `CAP_SYS_TIME` 能力（即 root 或专门授权）。但 `S91indigo` 启动脚本以普通用户启动 `indigo_server`，导致 `EPERM`。

**问题 B（次要）：系统时钟在 epoch，导致赤道仪时间也被污染**
当相机未联网时，系统时钟为 epoch（1970-01-01）。`X_UTC_SOURCE_HOST_ITEM` 为 "From Host" 时，INDIGO 每秒把 `time(NULL) ≈ 0` 推给赤道仪，导致赤道仪时间也变成 epoch，`meade_get_utc` 返回 -1（32位 `timegm` 溢出到 2070 年），Sync-from-Mount 拿到的时间值非法。

**📍 已做的代码修复**：

- 仓库：`git@github.com:eds-imagingtech/stargate.git`
- 分支：`fix/epoch-time-mount-corruption`(remote)
- 提交：`0e52553` `fix: prevent epoch time from corrupting mount UTC when system clock is unsynced`
- **修复内容**：
  - 自动同步守卫：`time(NULL) > 978307200`（2001-01-01）才推给赤道仪，避免 epoch 时间污染 mount
  - 显示守卫：mount 返回时间 < 2001 时，UTC Time 显示 ALERT 状态而非显示 1969-12-31
  - Sync-from-Mount 守卫：mount 时间 < 2001 时，提前报错 `"mount time may be uninitialized"`，不再传-1 给 `clock_settime`
  - 错误信息优化：区分 `EPERM`（"requires root"）和其他错误（"invalid time value"）

### 3.1 SmartWebServer + OnStepX OTA 固件升级系统

SmartWebServer 提供一个 web 页面，允许用户通过浏览器上传 `.bin` 固件，对 **MCU2（SmartWebServer 自身）** 和 **MCU1（OnStepX 主控板）** 进行 OTA 升级，无需 USB 下载器。

**整体架构：**

```
用户浏览器
    │  HTTP POST .bin 文件
    ▼
SmartWebServer（MCU2，ESP32 WiFi 模块）
    │ MCU2 自升级：直接用 ESP32 Update 库写 OTA 分区 → 重启生效
    │ MCU1 升级：先缓冲文件，再通过 UART 二进制协议转发
    ▼
OnStepX（MCU1，主控板）
    └── OtaReceiver 状态机接收 → ESP32 Update 库写 Flash → 重启生效
```

**升级 MCU2 自身（SmartWebServer.ino.bin）：**
- POST 到 `/update`，ESP32 `Update` 库流式写入 OTA 分区，写完重启
- 页面 JS 收到 `200 更新成功` 后等 8 秒自动刷新

**升级 MCU1（OnStepX.ino.bin）分两阶段：**

阶段一，浏览器 → MCU2（HTTP 上传）：
- POST 到 `/update_mcu1?size=<文件字节数>`（`?size` 参数是关键，`upload.totalSize` 在 ESP32 框架下不可靠，必须由 JS 从 `file.size` 传入）
- MCU2 收到第一块数据时立即发起握手，不等整个文件上传完
- 上传完成后页面收到 `MCU1_FORWARDING`，转为每 500ms 轮询一次 `/update_mcu1_status`

阶段二，MCU2 → MCU1（UART 二进制协议，115200 波特率）：

握手：
```
MCU2 → UART: ":FU00#"    # 唤醒 MCU1，进入 OTA trap
MCU1 → UART: "FR00#"     # MCU1 就绪，等待帧
```

握手后走二进制帧协议，帧格式：
```
[0x55][0xAA][CMD 1B][SEQ 2B][LEN 2B][PAYLOAD][CRC16 2B]
CRC16 覆盖范围：从 0x55 开始到 PAYLOAD 结束（含帧头）
```

| CMD | 方向 | 说明 |
|-----|------|------|
| `0x01` START | MCU2→MCU1 | 携带固件总大小、块大小、块总数 |
| `0x02` DATA | MCU2→MCU1 | 每块 512 字节，Payload = [seq 2B] + [data] |
| `0x03` END | MCU2→MCU1 | 携带 MD5（16B），触发 MCU1 校验并写入 |
| `0x04` ABORT | MCU2→MCU1 | 中止 |
| `0x10` ACK | MCU1→MCU2 | 确认，Payload 含被确认的 CMD + SEQ |
| `0x11` NACK | MCU1→MCU2 | 拒绝，含错误码 |

每帧有 3 次重试，超时 3 秒；每块发完后延迟 30ms 等 MCU1 写 Flash。

MCU1 `OtaReceiver` 状态机（`OnStepX/src/telescope/ota/OtaReceiver.cpp`）：
```
IDLE → (收到 START) → RECEIVING → (收到 END) → VERIFYING → COMPLETED → 自动重启
                                                           ↘ FAILED（MD5/Flash 错误）
```

状态查询接口：`GET /update_mcu1_status` → JSON
```json
{ "progress": 75, "complete": false, "success": false, "status": "传输数据", "error": "" }
```
MCU2 完成后将最终状态缓存 10 秒，防止 MCU2 重启前页面来不及轮询到结果。

**相关文件：**

| 仓库 | 文件 | 说明 |
|------|------|------|
| SmartWebServer | `src/pages/update/OTA.cpp` | Web 页面 HTML、`/update` 和 `/update_mcu1` 路由、状态查询接口 |
| SmartWebServer | `src/pages/update/OtaForwarder.cpp/.h` | MCU2 侧 UART 转发逻辑（握手 + 分帧 + 重试） |
| SmartWebServer | `src/pages/update/OtaTrap.cpp` | MCU2 进入 OTA 等待模式（被 MCU1 刷时用） |
| SmartWebServer | `src/pages/update/UartProtocol.h` | 帧格式常量、错误码、Payload 结构体（MCU1/MCU2 共用） |
| OnStepX | `src/telescope/ota/OtaReceiver.cpp/.h` | MCU1 OTA 接收状态机 + Flash 写入 |
| OnStepX | `src/telescope/ota/OtaTrap.cpp` | MCU1 进入 OTA 等待模式（收到 `:FU00#` 后调用） |
| OnStepX | `src/telescope/ota/UartProtocol.cpp/.h` | 与 MCU2 共用的帧协议实现（CRC16、帧构建） |

**⚠️ 注意事项：**
- OTA 中断电会导致 MCU 变砖，需用 USB 下载器（esptool）救砖
- 刷 MCU1 时 MCU2 必须已正常启动并连上 WiFi，否则没有 web 入口

### 3.2 stargate Imager 页面 Auto Focus HFD U-curve 曲线

在 INDIGO web 的 Imager 页面，在页面下方实时展示一个 **HFD Focus Analysis** 面板，用 U 形曲线可视化对焦过程，帮助用户判断对焦质量。

**功能描述：**

- 横轴：Focuser Position（对焦电机步进位置）
- 纵轴：HFD（Half Flux Diameter，半通量直径，越小代表星点越锐利）
- 每采到一个新位置的 HFD 数据，就在图上打一个散点，同时实时拟合二次抛物线（最小二乘法），曲线最低点即为最优对焦位置
- 面板中同时展示两组数据：Device Best（实际焦点位置/HFD）和 Min HFD Calc（从抛物线计算出的最优位置）

**数据流：**

```
INDIGO 后端（indigo_agent_imager.c）
    │ WebSocket 推送 AGENT_IMAGER_FOCUS_STATISTICS 属性
    │ 含 FOCUS_POSITION、HFD、BEST_FOCUS_DEVIATION 字段
    ▼
imager.html → handleHfdStatsUpdate()
    │ BEST_FOCUS_DEVIATION ≥ 90%：采样阶段，追加散点到 hfdPoints[]
    │ BEST_FOCUS_DEVIATION < 90%：结果阶段，更新 Device Best 面板
    ▼
refreshHfdUI() → calculateHfdQuadratic() → Chart.js 渲染
```

**实现文件：**

| 文件                                                  | 说明                                                                                          |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| `indigo_server/resource/imager.html`                | 图表 HTML 结构、Chart.js 初始化、`handleHfdStatsUpdate()`、`refreshHfdUI()`、`calculateHfdQuadratic()` |
| `indigo_server/resource/chart.min.js`               | Chart.js 库（本地化引入，避免依赖 CDN）                                                                  |
| `indigo_drivers/agent_imager/indigo_agent_imager.c` | 后端：autofocus 采样步数（focus stacking 改为 1 步/位置）                                                 |

**历次修改：**

| commit/PR | 时间 | 描述 |
|-----------|------|------|
| commit `4addf27` | 2026-01-14 | 初始实现：新增 HFD Focus Analysis 面板、Chart.js 图表、二次抛物线拟合、计时器 |
| PR `#5` (`a10c4cb`) | 2026-01-20 | focus stacking 改为 1（每个位置只拍 1 张，加快对焦速度） |
| PR `#12` (`adc6a10`) | 2026-03-23 | 修复三个 bug（见下方） |

**PR #12 修复的 bug（重要）：**

1. **浮点精度问题（catastrophic cancellation）**：当 focuser position 在 16000–17000 范围时，法方程矩阵中的 Σx⁴ 达到 ~8×10¹⁷，用 Cramer's rule 做差时 float64 的 15~16 位有效数字全部被消耗，二次项系数趋近于零，拟合曲线退化成近似水平线。修复方法：拟合前先对 x 做中心化（`x' = x - mean(x)`），矩阵量级从 10¹⁷ 降到 10⁸。

2. **曲线起始显示时机**：之前 HFD 面板是按下 Auto Focus 后才 `show()`，改为默认始终显示，让用户随时能看到面板。

3. **所有点位置相同时清除旧曲线**：若重新对焦后所有采样点的 x 坐标完全相同（极端情况），之前的曲线不会被清除，修复为此时主动清空 `datasets[0].data`。

**⚠️ 注意事项：**
- Chart.js 从 CDN 改为本地 `chart.min.js`（相机无外网）——如果图表不显示，先确认该文件存在于 `indigo_server/resource/`
- `calculateHfdQuadratic` 返回的 `minX` 是中心化坐标系下的最优位置，已在函数内转回原始坐标（`minX = -fit.b / (2*fit.a) + xMean`）——如果改这段代码要格外注意坐标系一致性

## 5. 踩过的坑 / 经验总结

### 坑 1：OnStepX GPIO36 无法使用 INPUT_PULLUP
- **症状**：固件启动串口报 `gpio_pullup_en(85): GPIO number error (input-only pad has no internal PU)`，设备可能不稳定
- **原因**：ESP32 GPIO36 是 input-only 引脚，没有内部上拉电阻；DS3231 的 SQW/INT 输出是 open-drain，板子上有外部上拉，不需要内部 PULLUP
- **解决**：`Pins.MaxESP3.h` 中定义 `PPS_SENSE_INIT INPUT`，`PPS.cpp` 中改用 `PPS_SENSE_INIT` 而不是硬编码 `INPUT_PULLUP`
- **预防**：ESP32 各 GPIO 的能力差异大，修改引脚相关代码前先查 ESP32 datasheet 或原理图

### 坑 2：OnStepX GitHub Actions tag 触发规则不完整
- **症状**：push 纯数字 tag（如 `1.0.1`）不触发 release Actions，带字母后缀（如 `1.0.1a`）才触发
- **原因**：`.github/workflows/build-release.yml` 的 tag pattern 是 `[0-9]*.[0-9]*.[0-9]*[A-Za-z]*`，强制要求末尾有字母
- **解决**：将 pattern 改为 `[0-9]*.[0-9]*.[0-9]*` 或 `v[0-9]*` 即可
- **预防**：设置 Actions 触发规则后，用不同格式的 tag 各测一次

### 坑 3：INDIGO UTC Time 每秒轮询覆盖用户输入
- **症状**：在 indigo web 的 Mount LX200 → Site → UTC Time 输入时间后，过一会儿时间自动被覆盖
- **原因**：INDIGO 框架每秒从本机系统时钟读取当前时间并推送到 UTC_TIME 属性，用户手动输入的值会立即被覆盖
- **解决**：实际使用时不需要修改 UTC Time（相机系统时间是准确的），只需要修改 UTC Offset 来适配不同时区
- **预防**：理解 UTC Time 字段的实际含义是"从 host 推给 OnStep 赤道仪的时间"，不是"相机自身的系统时间"

### 坑 4：stargate 在 seeing 中作为 submodule
- **症状**：修改了 `stargate` 并 push，但 `seeing` 项目中的 stargate 还是旧版本
- **原因**：`seeing` 以 submodule 方式引用 `stargate`，stargate 更新后 seeing 的 submodule 指针要手动更新
- **解决**：在 `seeing` 目录下 `cd stargate && git pull && cd .. && git add stargate && git commit -m "update stargate submodule"`
- **预防**：每次 stargate 有重要更新后，记得同步 seeing 的 submodule 指针
